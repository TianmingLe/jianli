# BorderGlow 集成到 Q&A 组件设计

## 目标

将 React Bits 的 `BorderGlow` 组件集成到现有 `Q&A` 页面中，并按用户确认的范围应用到两层结构：

- 第一层：每个主问题块
- 第二层：第三个问题下的场景化价值卡片

集成后的效果需要满足以下要求：

- 保留当前 `Q&A` 的内容结构、文案层级和进入动画
- 使用 `BorderGlow` 替换现有较普通的边框/背景卡片表现
- 让主问题块与内部场景卡片都具有边缘辉光交互
- 避免双层辉光叠加后页面过于花哨，保持当前深色科技感

## 当前代码现状

### 页面层

`src/pages/QA.tsx` 负责：

- 页面头部 `PageHeader`
- `Q&A` 内容组件 `QandA`
- 已实现的背景滚动、模糊、暗化和底部黑色渐层

这部分不需要因 `BorderGlow` 集成而改动结构。

### 内容层

`src/components/QandA.tsx` 负责：

- 渲染章节标题
- 循环渲染每个主问题块
- 在第三个问题中渲染场景化价值卡片
- 使用 `framer-motion` 给主问题块和场景卡片做进入动画

当前样式特征：

- 主问题块使用 `border-t` 做分隔
- 第三个问题的场景卡片使用普通边框和半透明背景
- 尚未使用独立的可复用卡片容器组件

## 集成范围

本次仅做 `Q&A` 区域的 `BorderGlow` 接入，不扩散到其他页面或组件。

### 包含

- 新增 `BorderGlow` 组件源码
- 新增 `BorderGlow.css`
- 在 `QandA.tsx` 中接入 `BorderGlow`
- 根据当前页面视觉风格调整 `BorderGlow` 参数
- 在主问题块和 Q3 子卡片两个层级上分别应用不同强度的辉光配置

### 不包含

- 重写 `Q&A` 文案数据结构
- 重做 `QA` 页面背景逻辑
- 大规模重构整个 `Q&A` 组件
- 扩展到 `Home`、`Projects` 等其他页面

## 方案选择

### 方案 A：直接在现有结构中接入 BorderGlow

做法：

- 保留 `motion.article` 外层
- 在主问题块内部增加 `BorderGlow`
- 在 Q3 卡片内部再包一层较轻量的 `BorderGlow`

优点：

- 改动范围小
- 风险最低
- 不破坏现有动画和内容结构
- 最符合用户已确认的“两层都要”需求

缺点：

- `QandA.tsx` 结构会比现在更长一点

### 方案 B：先拆分子组件再接入 BorderGlow

做法：

- 先把主问题块和场景卡片拆成独立子组件
- 再在各子组件中接入 `BorderGlow`

优点：

- 结构更清晰
- 更利于以后复用

缺点：

- 超出本次任务必要范围
- 增加实现成本与回归风险

### 方案 C：用 BorderGlow 完全替换当前视觉容器

做法：

- 把现有边框、背景、分隔逻辑尽量去掉
- 由 `BorderGlow` 主导整个视觉外观

优点：

- 视觉变化最明显

缺点：

- 容易破坏当前页面节奏
- 双层辉光可能过强
- 调参成本最高

## 最终采用方案

采用方案 A。

原因：

- 用户已经确认希望在两个层级都应用 `BorderGlow`
- 现有 `Q&A` 组件结构清晰，适合局部增强
- 该方案最容易保留当前已完成的背景氛围和动效

- 支持 React 项目直接导入使用

实现说明：

- 导入同目录 `BorderGlow.css`

#### `src/components/BorderGlow.css`

职责：

- 提供 `BorderGlow` 所需的全部类名和伪元素样式

说明：

- 直接收纳该组件提供的 CSS
- 仅在必要处做极小调整，避免和现有全局样式冲突

### 修改文件

#### `src/components/QandA.tsx`

调整点：

- 引入 `BorderGlow`
- 每个主问题块外部保留 `motion.article`

- 第三个问题下的每张场景卡片继续保留 `motion.div` 进入动画，再在其内部加入第二层 `BorderGlow`

预计不修改功能逻辑。

## 结构设计

### 主问题块层级

结构将调整为：

- `motion.article`
- `BorderGlow`
- 问题编号、问题标题、回答段落
- 如果有场景卡片，则在内容末尾继续渲染子卡片网格

说明：

- `motion.article` 继续负责进入动画和锚点定位
- `BorderGlow` 负责视觉容器效果
- 原有 `border-t` 会移除或弱化，因为辉光容器本身已承担容器边界表达

### Q3 场景卡片层级

结构将调整为：

- 场景卡片网格容器
- `motion.div`
- `BorderGlow`
- 卡片标题、痛点、价值

说明：

- 内层卡片的辉光参数必须更克制
- 保留当前信息层次，不因视觉效果影响阅读顺序

## 参数策略

### 主问题块 BorderGlow

建议参数方向：

- `borderRadius`: 较大，匹配当前页面较宽松的块级容器
- `glowRadius`: 中等
- `glowIntensity`: 中低
- `fillOpacity`: 偏低
- `edgeSensitivity`: 中等
- `animated`: 关闭
- `backgroundColor`: 深色，贴合现有背景体系
- `colors`: 使用偏紫、偏粉、偏蓝的组合，但饱和度控制在中等范围

目标：

- 鼠标靠近边缘时有明显反馈
- 非悬停状态下仍保持页面整体克制

### Q3 子卡片 BorderGlow

建议参数方向：

- `borderRadius`: 小于主问题块
- `glowRadius`: 更小
- `glowIntensity`: 低于主问题块
- `fillOpacity`: 更低
- `edgeSensitivity`: 略高，让小卡片更容易触发
- `animated`: 关闭

目标：

- 保持精致感
- 避免和外层主块形成过强的视觉竞争

## 动画与交互兼容

### 与 framer-motion 的关系

保留现有 `motion.article` 与 `motion.div`。

原则：

- `framer-motion` 负责滚动进入时的位移和透明度动画
- `BorderGlow` 负责悬停时的边缘辉光交互

这样可以避免把两套职责混在同一层容器上，降低冲突概率。

### 指针事件

`BorderGlow` 的指针跟踪依赖 `onPointerMove`。

因此：

- 交互层必须包裹真实可悬停的卡片区域
- 不能把需要感应鼠标的内容全部放在不会接收事件的父层之外

## 样式兼容策略

为避免与现有 Tailwind 类冲突，按以下原则处理：

- 保留文字颜色、字号、间距等排版类
- 移除会与 `BorderGlow` 重复的普通边框背景类
- 如果当前容器已有 `backdrop-blur`、半透明背景等样式，则根据实际效果决定保留还是简化
- 不修改全局 `index.css`，除非组件 CSS 必须做最小兼容修正

## 风险与处理

### 风险 1：双层辉光过强

处理：

- 外层主问题块使用更低填充和中低亮度
- 内层场景卡片进一步降低强度与范围

### 风险 2：进入动画与辉光层叠导致闪烁

处理：

- 让 `motion` 容器包住 `BorderGlow`
- 不在初始挂载时启用 `animated sweep`

### 风险 3：布局被新容器撑开

处理：

- 控制 `BorderGlow` 的内层结构只包裹原有内容
- 根据实际效果微调 `p-*` 和网格间距

### 风险 4：CSS 伪元素层级遮挡文本

处理：

- 保留组件源码中的层级定义
- 如有需要，仅微调 `z-index` 和 `overflow` 相关样式

## 验证计划

实现后需要完成以下检查：

- 主问题块在桌面端鼠标靠近边缘时能正常出现辉光
- Q3 子卡片在桌面端也能正常跟随指针出现辉光
- 文本可读性不下降
- 双层容器不会造成明显视觉噪音
- `framer-motion` 的进入动画仍然正常
- 页面背景滚动模糊逻辑不受影响
- 最近修改文件无明显诊断错误
